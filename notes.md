# AI GARP project notes

## Purpose
Collect and maintain a daily panel of Best Buy prices + metadata for
revealed-preference (GARP) analysis.

## Key decisions

- **2026-04-20 — Storage backend: Dropbox, App-folder scope.**
  App name: `AIGarpDataPipeline`. Written via Dropbox API directly from
  GitHub Actions; syncs down to local machine via Dropbox desktop client
  when laptop is on. App-folder scope (not Full Dropbox) limits blast
  radius if credentials leak.

- **2026-04-20 — Scheduler: single GitHub Actions workflow, 4×/day.**
  Cron: `0 0,6,12,18 * * *` UTC. No Windows scheduled task for ingestion
  (laptop uptime cannot gate the pipeline). Both prices and metadata
  are fetched in the same sweep, since Best Buy's Products API returns
  both in one call — splitting would double quota usage for no gain.

- **2026-04-20 — Storage format: partitioned parquet.**
  Prices: append-only, one file per run, partitioned by date/hour.
  Metadata: SCD2 diff-only via content-hash index (only rows that
  changed get written to history). Run weight stays constant forever;
  no read-append-reupload cycle.

- **2026-04-20 — Scope: whole Best Buy catalog (~1M SKUs).**
  User wants discovery breadth for GARP analysis, not a restricted
  panel.

- **2026-04-20 — Windows scheduled tasks: not used for ingestion.**
  May be reintroduced in Phase 8 for heavy local post-processing only.

## Phase 2 audit findings (2026-04-21)

Full audit report: `audit_report.json`.

- **Scope: active catalog only.**
  - `active=true`: **56,347 SKUs** (this is the default with no filter)
  - `active=false`: 2,416,921 SKUs (discontinued archive; excluded)
  - Our pipeline queries `/products(active=true)` explicitly.
- **Schema: 182 fields available** with `show=all`. Curated down to 15
  price fields + 37 metadata fields (see `bbpipeline/config.py`).
  Excluded: all image fields, URLs, HTML description variants,
  carrier/plan/protection-plan data, marketing/recommendation fields.
- **Pagination**: `pageSize=100` max, clean disjoint pages,
  `totalPages=564` for the active catalog. Full sweep = 564 requests.
- **Rate limit**: docs say 5 req/s, **observed ~1 req/s in practice**.
  At 1.2s spacing we had zero 403s. Set
  `BESTBUY_MIN_INTERVAL_SECONDS = 1.2` in config.
- **`itemUpdateDate` filter**: works. 27,116 SKUs (~48%) updated in the
  last 7 days — high churn, but hash-index diff will catch only the
  structural-metadata changes, not the behavioral ones.
- **Latency**: 1.08–1.38s per request, median 1.14s.
- **Full sweep wall time**: ~11 minutes at 1.2s pacing. Well inside the
  6h GitHub Actions cap; easy headroom for 4×/day.
- **Per-day API usage**: 4 × 564 = **2,256 requests/day**, plus a small
  number for health checks. Previously we feared the 50k/day quota
  (if that's still enforced); we're using <5%.
- **Daily quota in 2026**: not yet explicitly verified, but the
  operational policy page still exists. Our usage is far below any
  reasonable quota.
